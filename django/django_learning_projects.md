# Django 学习推荐项目 · 部署方向

本篇整理：**适合练手的 Django 项目线**，以及学完基础后如何把项目做成「**可部署实战**」（Nginx + Gunicorn + MySQL，Docker 可选），并列举适合做「最后一公里」上线的**系统类型**。

---

## 一、按难度推荐的五个练手方向

目标是「做出能跑的项目」，而不是只听概念。

### 1. Todo 清单（入门必做）

- **练**：`Model`、CRUD、URL、模板、`Form`/`POST`
- **目标**：新建 / 编辑 / 删除 / 完成状态的切换都做通

### 2. 博客系统（经典综合）

- **练**：文章与分类标签、分页、（可选）富文本、后台 `admin`
- **目标**：列表页、详情页、发布与管理、简单搜索或筛选

### 3. 用户中心 + 权限

- **练**：注册登录、会话/权限、`LoginRequiredMixin`、`User` 与资源归属（只能改自己的数据）
- **目标**：分层角色（访客 / 注册用户 / 管理员）能落在真实业务直觉上

### 4. 简化电商（偏实战）

- **练**：商品、购物车、订单、库存与事务、`F()` / 原子扣减、结账流程
- **目标**：可走通「下单 → 订单明细 →（可选）支付沙箱」，理解并发下库存的注意点  
- **与本仓库**：`AiBasic/simplified_commerce` 为该方向的 Docker 骨架示例可参考

### 5. REST API（Django + DRF）

- **练**：`DRF`、`Serializer`、`ViewSet`、JWT/Session、分页与过滤、接口权限与 CORS
- **目标**：把博客类或 Todo 改成前后端分离的 API，利于贴近公司协作方式

---

## 二、学习路线建议（很实用）

先做 **Todo + 博客** 打基础和 Django 全流程；再上 **用户与权限**；然后学 **DRF**（若职业规划偏 Web/API）；最后收尾做一章里的 **「可部署项目」**（见下文），打通运维与发布认知。

---

## 三、「可部署项目」指什么

不是某一种固定业务，而是你的应用以**接近生产**的方式跑起来：

- **Nginx**：反向代理、（通常）托管静态资源，有时限流与 HTTPS 终止
- **Gunicorn**：跑 Django 的 WSGI 应用进程
- **MySQL**：持久化数据（也可用 PostgreSQL，思路类似）
- **Docker / Compose（可选）**：容器化应用与数据库，环境一致、上云迁移方便

没有 Docker 也可以在一台 Linux 上直接装 Nginx + Gunicorn + MySQL；加上 Docker 主要练 **镜像、compose、环境变量与卷**。

## 四、适合做「可部署收官」的系统类型举例

下面几类都**足够真实**，又适合作为个人作品集里「能上线」的一条。

### 1. 内容类：博客 / 文档站 / 小型 CMS

- **为何合适**：文章、评论、用户、后台管理；MySQL 自然；静态资源多，正好练 Nginx 与 `collectstatic`。
- **部署侧重**：媒体文件策略、HTTPS、数据库备份。

### 2. 工具类：短链、图床、在线剪贴板、带过期时间的文件分享

- **为何合适**：逻辑清晰，但有访问压力、限流、存储路径等运维问题。
- **部署侧重**：Nginx 限流或缓存头、Gunicorn worker 数、大文件与磁盘或对象存储。

### 3. 协作类：团队 Wiki、任务看板、内部工单（简化版）

- **为何合适**：多用户、权限、状态流，接近公司内部系统。
- **部署侧重**：安全相关 HTTP 头、迁移与多环境配置（`DEBUG=False`、密钥走环境变量）。

### 4. 业务类：简化电商、课程/活动报名 + 支付回调（沙箱）

- **为何合适**：订单、库存、事务、回调 URL，**外网可访问**才有完整意义。
- **部署侧重**：事务与幂等、日志、（进阶）只读从库等可后续再加。

### 5. API 服务：给前端或移动端的 REST 后端（DRF）

- **为何合适**：与「前后端分离 + 独立部署」一致。
- **部署侧重**：JWT、CORS、Nginx 反代到 Gunicorn、API 限流。

### 怎么选

- 时间紧：**博客 + 登录 + 后台** 或 **Todo/看板 + 多用户**。
- 想更贴近工作：**简化电商** 或 **工单/报名 + 回调沙箱**。
