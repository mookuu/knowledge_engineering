# 阶段二：技术架构设计

## 概述

在 PRD 明确之后，进入技术架构设计阶段。此阶段决定项目的技术选型、系统模块划分、数据模型设计和接口规范。AI 在此阶段的作用是**辅助方案对比、生成设计文档、生成骨架代码**，**关键决策由人做出**。

## 核心交付物

| 交付物 | 说明 | AI 可参与程度 |
|---|---|---|
| 技术选型文档 | 语言/框架/数据库/中间件选择与理由 | 高 — 辅助对比分析 |
| 系统架构图 | 模块划分、依赖关系、通信方式 | 中 — 辅助 Mermaid 绘图 |
| 数据模型 | ER 图、表结构、字段定义 | 高 — 从 PRD 推导 |
| API 设计 | REST/GraphQL 接口定义、请求/响应格式 | 高 — 辅助生成 OpenAPI 规范 |
| 目录结构 | 项目文件和模块组织方式 | 高 — 根据框架生成模板 |
| 风险评估 | 技术风险点及缓解策略 | 中 — 辅助识别 |

## AI 辅助工作流

```
PRD / 需求文档
    ↓
AI 辅助 → 技术选型方案对比（列出选项 + 优缺点）
    ↓
人工决策 → 确定技术栈
    ↓
AI 辅助 → 生成系统架构描述 + 架构图（Mermaid）
    ↓
AI 辅助 → 推导数据模型（从用户故事提取实体）
    ↓
AI 辅助 → 生成 API 设计（OpenAPI/Swagger）
    ↓
AI 辅助 → 生成项目骨架目录结构
    ↓
人工评审 → 确认设计
```

### 步骤 1：技术选型

让 AI 对比技术方案，但**不要直接让 AI 决定**。更好的方式：

```
# 好的 prompt
请对比以下方案做前端项目：
- 方案 A: React + Vite + TypeScript
- 方案 B: Vue 3 + Vite + TypeScript
从以下维度对比：生态成熟度、团队学习成本、性能、
类型安全、构建速度、CI/CD 集成方便度。

# 不好的 prompt（直接把决策权给 AI）
我应该用什么前端框架？
```

**选型时应考虑的维度**：
- 团队熟悉度（AI 不知道你团队会什么 — 主动告诉它）
- 生态成熟度（社区活跃度、包数量、文档质量）
- 性能要求（高并发？实时性？）
- 部署环境（云原生？边缘计算？嵌入式？）
- 维护成本（长期维护 vs 一次性项目）
- 集成需求（需要和哪些现有系统对接）

### 步骤 2：架构设计

AI 擅长从需求描述生成架构草稿。推荐使用 **C4 模型** 分层描述：

```
Level 1: 系统上下文图（System Context）
  └── 我们的系统与外部系统（用户、第三方 API、数据库等）的关系

Level 2: 容器图（Container）
  └── 系统由哪些应用/服务组成（Web App、API Server、DB、Cache 等）

Level 3: 组件图（Component）
  └── 每个容器内部的模块划分

Level 4: 代码（Code）
  └── 类/接口/文件结构（这一层在开发阶段细化）
```

**AI 提示示例**：
```
项目需求：{PRD 摘要}
技术栈：React + FastAPI + PostgreSQL
请用 C4 模型的 Level 1 和 Level 2 描述架构，
并用 Mermaid 生成架构图。
```

### 步骤 3：数据模型设计

AI 能从用户故事中提取实体和关系：

```
用户故事：
- 用户可以创建文章
- 文章属于一个分类
- 用户可以评论文章
- 每篇文章可以有多个标签

请推导数据模型，用 Mermaid ER 图表示，
包含字段名、类型、主键、外键和索引建议。
```

**AI 辅助数据建模的注意事项**：
- AI 容易设计过范式化（过度拆分表）或欠范式化（大宽表）
- 需要人工复核：字段类型、长度、约束是否合理
- 要求 AI 补充索引建议和查询模式分析

### 步骤 4：API 设计

```
根据以下数据模型生成 RESTful API 设计：
{数据模型}

要求：
- 使用 OpenAPI 3.0 格式
- 包含每个接口的请求/响应格式
- 标注认证方式
- 标注错误码
```

### 步骤 5：项目骨架生成

```
请为一个 {框架名} 项目生成推荐的目录结构：
功能模块：{模块列表}
数据存储：{数据库类型}
```

AI 可以生成初始目录，但你应根据项目实际情况调整。

## 设计评审检查清单

- [ ] 技术选型是否有明确理由（不只用"流行"作为理由）
- [ ] 架构是否覆盖所有功能模块
- [ ] 数据模型是否满足所有用户故事
- [ ] API 设计是否 RESTful / 符合团队规范
- [ ] 安全机制是否提前考虑（认证、授权、数据加密）
- [ ] 扩展性是否有考虑（未来增加新模块的难度）
- [ ] 是否有考虑非功能性需求（性能、可用性、灾备）
- [ ] 目录结构是否清晰、符合项目约定

## 常用 AI Prompts

### 架构方案对比
```
请对比 {方案A} 和 {方案B}，从以下维度展开：
1. 学习曲线
2. 性能基准
3. 社区生态
4. 可维护性
5. 典型坑点
```

### 生成 Mermaid 架构图
```
请用 Mermaid 生成 {系统名称} 的架构图，
包含：用户端、负载均衡、应用服务、数据库、缓存、消息队列。
```

### 设计评审
```
请评审以下架构设计，指出潜在风险：

{架构描述}
```

| **Excalidraw** | 白板绘图 | 免费手绘风格架构图 |

| **PlantUML** | UML 图 | C4 模型友好 |

## 注意事项 / 陷阱

- ⚠️ **AI 倾向于过度工程化** — AI 容易推荐微服务、消息队列、缓存等"高级"方案，对于简单项目可能过度设计。主动约束：给 AI 设定预算或团队规模上下文
- ⚠️ **技术选型缺乏团队上下文** — AI 不知道你团队擅长的技术栈。你必须主动提供团队背景
- ⚠️ **架构决策不记录 ≈ 没做** — 关键决策及其理由应存档（使用 `remember project`），供后续阶段和新人查阅
- ⚠️ **AI 生成的目录结构可能不符合框架约定** — 生成后对照官方推荐结构检查
- ⚠️ **性能预估不可信** — AI 给出的"QPS预估""延迟估算"基于训练数据中的一般情况，不是你的实际场景。不要用 AI 的估算做容量规划
- ✅ **最佳实践**：让 AI 从反面挑战你的设计（"Tell me why this architecture will fail"）
