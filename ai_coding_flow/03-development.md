# 阶段三：编码实现

## 概述

编码实现阶段是将设计转化为可运行代码的核心环节。AI Coding Agent（如 Reasonix、Cursor、Copilot）在此阶段作用最显著：**代码生成、补全、解释、重构、调试**。核心原则是"AI 编码，人工负责"。

## 核心交付物

| 交付物 | 说明 | AI 可参与程度 |
|---|---|---|
| 功能代码 | 实现需求的代码 | 高 — 从任务/描述生成 |
| 单元测试 | 验证代码正确性 | 高 — 见测试阶段 |
| 代码注释 | 关键逻辑说明 | 高 — AI 自动生成/更新 |
| 类型定义 | TypeScript/Python 类型注解 | 高 — AI 自动补充 |
| 数据库迁移 | 数据库 schema 变更脚本 | 中 — 辅助生成 |

## AI 辅助工作流

```
任务说明 / 设计文档
    ↓
(推荐) TDD 流程：先写测试 → 再实现
    ↓
AI 辅助 → 生成测试用例（单元/集成）
    ↓
AI 辅助 → 生成实现代码（最小可通过测试）
    ↓
人工审查 → 确认逻辑正确
    ↓X?
AI 辅助 → 重构优化（如有必要）
    ↓
运行测试 → 全部通过
    ↓
提交版本管理
```

### 几种主流工作模式

#### 模式 1：TDD（测试驱动开发）

```
1. 编写测试（描述期望行为）
2. 运行测试 → 预期失败（Red）
3. AI 生成最少代码让测试通过（Green）
4. 重构优化（Refactor）
5. 重复
```

**AI 配合方式**：
```
Task: 实现一个函数 isPalindrome(s: str) -> bool
请先生成 pytest 测试用例（覆盖：空字符串、单字符、
偶数长度、奇数长度、带空格、大小写混合），
然后再给出实现代码。
```

#### 模式 2：描述驱动（Describe → Generate）

```
请实现以下功能：
- 用户注册接口 POST /api/register
- 接收参数：email, password, name
- 密码需要 bcrypt 加密
- 返回：user_id, token
使用 FastAPI + SQLAlchemy + PostgreSQL
```

**适合**：从零开始的新模块、CRUD 操作、模板化代码。

#### 模式 3：迭代修改（Iterative Edit）

```
当前代码：
{现有代码片段}
需要修改：
{具体改动描述}
```

**适合**：存量项目的小范围改动、Bug 修复。

## AI 编码最佳实践

### 1. 提供充分上下文

AI 对完全陌生的代码库理解有限。每次请求前提供：

| 上下文类型 | 示例 |
|---|---|
| **项目背景** | "这是一个 Django REST Framework 项目，使用 PostgreSQL" |
| **文件结构** | "相关文件在 app/services/ 和 app/models/" |
| **现有模式** | "项目使用 repository pattern，Model 不直接暴露给 View" |
| **命名约定** | "变量使用 snake_case，URL 使用 kebab-case" |

### 2. 使用 /new 做任务切换

Reasonix 等 Agent 的上下文窗口有限。大幅度切换任务时：

```
# 完成一个模块后
- 将关键信息 remember（项目约定、本次变更说明）
- 执行 /new 清空上下文
- 在新会话中恢复上下文
```

### 3. 小步提交，频繁验证

- 每次改动范围控制在单个功能/单个文件
- 每步完成后运行测试/lint
- 不要一次让 AI 改太多文件（易产生冲突/遗漏）

### 4. 使用统一的项目规范

AI 需要知道项目的编码规范才能生成风格一致的代码：

| 规范类型 | 应记录的内容 |
|---|---|
| **代码风格** | ESLint/Prettier/PEP8/Ruff 配置 |
| **命名规范** | 文件名、变量名、类名、URL 路径 |
| **组件模式** | 使用函数组件还是类组件？ |
| **状态管理** | Redux/Zustand/Context? |
| **错误处理** | 统一错误响应格式？ |
| **日志规范** | 日志级别、格式、敏感信息脱敏 |

> 将这些规范用 `remember project` 保存，或写入项目根目录的 `CODING_GUIDE.md`

## AI 编码的常见场景与提示

### 新功能开发
```
请按以下步骤实现 {功能名称}：
1. 先在 models.py 中添加数据模型
2. 创建 serializer
3. 实现 view/endpoint
4. 添加 URL 路由
每步完成后请我确认再继续。
```

### Bug 修复
```
Bug: 用户点击提交按钮后页面无响应
日志：{粘贴错误日志}
相关代码：
{粘贴代码}
请分析原因并给出修复方案。
```

### 代码重构
```
请重构以下代码，目标是提高可读性和可测试性：
{代码}
约束：
- 保持对外接口不变
- 提取纯函数
- 减少嵌套深度
- 使用当前项目已有的工具函数
```

### 代码解释（新接手项目）
```
请解释以下代码块的逻辑：
{代码}
重点说明：输入输出格式、边界处理、外部依赖。
```

## 注意事项 / 陷阱

- ⚠️ **AI 生成的代码有隐含 bug** — 边界条件、并发安全、资源泄露是最常见遗漏点。务必执行 review 和测试
- ⚠️ **幻觉 API / 库** — AI 可能虚构不存在的函数或方法。遇到无法运行的代码先查官方文档
- ⚠️ **上下文泄露** — 长时间会话中，AI 可能混淆不同任务/模块的上下文。关键节点使用 `/new` 重置
- ⚠️ **不要信任 AI 的安全意识** — AI 默认不关注安全。SQL 注入、XSS、权限绕过需要人工审核
- ⚠️ **过度生成** — AI 倾向于生成超出需求的代码（如额外端点、多余字段）。严格按 PRD/设计实现
- ⚠️ **许可证风险** — AI 可能生成 GPL/AGPL 等强传染协议的代码片段。商业项目需要额外审查
- ✅ **最佳实践**：Git 提交前运行 `git diff` + AI review，确保每行代码都是需要的
