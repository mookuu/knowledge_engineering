# 阶段一：PRD / 需求分析

## 概述

PRD（Product Requirements Document，产品需求文档）是整个 AI-SDLC 的起点。高质量的 PRD 决定了后续所有阶段的效率和质量。AI 在此阶段的主要作用是**辅助分析、撰写初稿、拆解任务**，而非替代产品经理做决策。

## 核心交付物

| 交付物 | 说明 | AI 可参与程度 |
|---|---|---|
| PRD 文档 | 功能描述、业务规则、边界条件 | 高 — 辅助生成初稿和细化 |
| 用户故事 | 从用户视角描述需求 | 高 — 从 PRD 自动拆解 |
| 验收标准（Acceptance Criteria） | 可验证的完成条件 | 中 — 辅助编写 |
| 任务拆解（Task Breakdown） | 开发任务列表 | 高 — 从用户故事拆分 |
| 流程图/用户旅程 | 用户操作路径 | 中 — 辅助绘制（Mermaid） |

## AI 辅助工作流

```
用户提供原始需求（一句话/文档/会议纪要）
    ↓
AI 辅助 → 生成 PRD 初稿（结构化格式）
    ↓
人工审核 → 补充业务细节、修正理解偏差
    ↓
AI 辅助 → 拆解为用户故事 + 验收标准
    ↓~
AI 辅助 → 拆分为开发任务（含优先级）
    ↓
人工确认 → 进入下一阶段
```

### 步骤 1：原始需求采集

- 用户提供：口头描述、会议纪要、竞品分析、Issue 描述等
- **AI 建议**：对模糊需求主动提问澄清（5W1H：Who / What / When / Where / Why / How）
- **最佳实践**：原始输入越具体，AI 产出的 PRD 质量越高

### 步骤 2：AI 生成 PRD 初稿

使用预设的 PRD 模板让 AI 生成初稿。推荐的 PRD 结构：

```
# PRD: [功能名称]

## 1. 背景与动机
   - 为什么做这个功能
   - 解决什么用户问题

## 2. 目标与成功指标
   - 业务目标
   - 可量化的成功标准（OKR/KPI）

## 3. 功能范围
   - In Scope（包含的功能）
   - Out of Scope（明确不包含的）

## 4. 用户故事
   - As a [用户角色], I want to [目标], so that [收益]

## 5. 验收标准
   - Given... When... Then...

## 6. 边界条件/异常场景
   - 网络异常、空数据、权限不足等

## 7. 非功能性需求
   - 性能、安全、可访问性、国际化

## 8. 相关文档/参考
```

### 步骤 3：AI 辅助拆解

将 PRD 拆解为可执行的任务：

```
PRD: 用户登录功能
  ├── 用户故事 1: 用户可以使用邮箱+密码登录
  │   ├── 任务 1.1: 设计登录页 UI
  │   ├── 任务 1.2: 实现登录 API
  │   ├── 任务 1.3: 实现 JWT Token 管理
  │   └── 任务 1.4: 实现登录状态持久化
  ├── 用户故事 2: 用户可以使用 Google OAuth 登录
  │   ├── 任务 2.1: 集成 Google OAuth SDK
  │   └── 任务 2.2: 处理 OAuth 回调
  └── 用户故事 3: 登录失败时给出友好提示
      ├── 任务 3.1: 错误码映射
      └── 任务 3.2: 前端错误展示
```

### 步骤 4：PRD 评审检查

- [ ] 每个用户故事是否有明确的"谁-要什么-为什么"
- [ ] 验收标准是否可用 Given/When/Then 格式表达
- [ ] 边界条件和异常场景是否覆盖
- [ ] 非功能性需求是否提及
- [ ] Out of Scope 是否明确

## 常用 AI Prompts

### 需求澄清
```
用户需求是：「{原始需求}」
请用 5W1H 列出需要澄清的问题，按重要性排序。
```

### 生成 PRD 初稿
```
请根据以下需求撰写 PRD：{需求描述}
格式要求：
- 包含背景、范围、用户故事、验收标准
- 用户故事用 As a/I want/So that 格式
- 验收标准用 Given/When/Then 格式
```

### 任务拆解
```
请将以下 PRD 拆解为开发任务（粒度：半天到一天）：
{PRD 内容}
按模块分组，标注依赖关系，估算复杂度（S/M/L）。
```

## 注意事项 / 陷阱

- ⚠️ **AI 容易编造需求** — AI 可能"脑补"不存在的功能。务必人工审核每一条用户故事是否真实业务需要
- ⚠️ **过度拆解** — 任务粒度过细（< 1 小时）反而增加管理成本。推荐粒度：半天 ~ 2 天
- ⚠️ **忽视非功能性需求** — AI 默认聚焦功能，需要主动提示其补充性能/安全/可访问性
- ⚠️ **需求蔓延** — Out of Scope 要写清楚，AI 容易被"再多加一个小功能"带偏
- ✅ **最佳实践**：PRD 完成后先做一轮 AI 辅助的"找漏洞"（Ask AI to find edge cases）
