# 阶段一：PRD / 需求分析

## 概述

PRD（Product Requirements Document，产品需求文档）是整个 AI-SDLC 的起点。高质量的 PRD 决定了后续所有阶段的效率和质量。AI 在此阶段的主要作用是**辅助分析、撰写初稿、拆解任务**，而非替代产品经理做决策。

## 核心交付物

| 交付物                          | 说明                                                                                     | 粒度                | AI 可参与程度            |
| ------------------------------- | ---------------------------------------------------------------------------------------- | ------------------- | ------------------------ |
| PRD 文档                        | 功能描述、业务规则、边界条件                                                             | —                   | 高 — 辅助生成初稿和细化  |
| 用户故事（功能级）              | 从用户视角描述需求 —— `As a [角色], I want [目标], so that [收益]`。**prd-writing 产出** | 一个功能/模块       | 高 — 从 PRD 自动拆解     |
| 验收标准（Acceptance Criteria） | 可验证的完成条件（Given/When/Then）                                                      | 每个用户故事 2~3 条 | 中 — 辅助编写            |
| 任务拆解（Task Breakdown）      | 开发任务列表，含前后端/测试拆分。**story-splitting 产出**                                | ≤ 2d / 任务         | 高 — 从用户故事拆分      |
| 流程图/用户旅程                 | 用户操作路径                                                                             | —                   | 中 — 辅助绘制（Mermaid） |

## AI 辅助工作流

```
用户提供原始需求（一句话/文档/会议纪要）
    ↓
① AI 辅助 (prd-writing) → 生成 PRD 初稿
    ↓
② 人工审核 → 补充业务细节、修正理解偏差
    ↓
③ AI 辅助 (prd-writing) → 拆解为功能级用户故事 + 验收标准
    ↓
④ AI 辅助 (story-splitting) → 拆分为可执行开发任务（≤ 2d 粒度）
    ↓
⑤ 人工确认 → 进入下一阶段
```

> **关键区分**: ③ 和 ④ 是两个不同粒度的拆分，对应两个不同 Skill，详见下方「prd-writing vs story-splitting」。

### 步骤 1：原始需求采集

- 用户提供：口头描述、会议纪要、竞品分析、Issue 描述等
- **AI 建议**：对模糊需求主动提问澄清（5W1H：Who / What / When / Where / Why / How）
- **最佳实践**：原始输入越具体，AI 产出的 PRD 质量越高

### 步骤 2：AI 生成 PRD 初稿

使用预设的 PRD 模板让 AI 生成初稿。推荐的 PRD 结构：

```
# PRD: [功能名称]

## 1. 背景与动机
   - 为什么做这个功能
   - 解决什么用户问题

## 2. 目标与成功指标
   - 业务目标
   - 可量化的成功标准（OKR/KPI）

## 3. 功能范围
   - In Scope（包含的功能）
   - Out of Scope（明确不包含的）

## 4. 用户故事
   - As a [用户角色], I want to [目标], so that [收益]

## 5. 验收标准
   - Given... When... Then...

## 6. 边界条件/异常场景
   - 网络异常、空数据、权限不足等

## 7. 非功能性需求
   - 性能、安全、可访问性、国际化

## 8. 相关文档/参考
```

### 步骤 3：prd-writing — 拆解为功能级用户故事

使用 `prd-writing` Skill 将 PRD 中的功能描述转化为 **功能级用户故事**。

> 每个用户故事 = 一个完整功能点，面向**产品/团队对齐需求**。

### 步骤 4：story-splitting — 拆分为开发任务

使用 `story-splitting` Skill 将每个功能级用户故事进一步拆解为 **可执行的开发任务**。

> 每个任务 ≤ 2d，可分配给单个工程师，面向**开发/测试排期**。

**举例 — 同一用户故事的两个粒度层级：**

```
PRD: 用户登录功能
  └── US-01 [prd-writing 产出] 用户可以使用邮箱+密码登录
       ├── 任务 1.1 [story-splitting 产出] 后端 — 实现登录 API（JWT 签发）
       ├── 任务 1.2 [story-splitting 产出] 后端 — 实现登录状态持久化（Session/Token）
       ├── 任务 1.3 [story-splitting 产出] 前端 — 设计登录页 UI
       ├── 任务 1.4 [story-splitting 产出] 前端 — 实现登录表单验证 + 错误提示
       └── 任务 1.5 [story-splitting 产出] 测试 — 登录流程集成测试
```

### prd-writing vs story-splitting

| 维度             | `prd-writing`                                  | `story-splitting`                    |
| ---------------- | ---------------------------------------------- | ------------------------------------ |
| **输入**         | 原始需求 / 已有 PRD                            | PRD 中的**功能级用户故事**           |
| **产出**         | 功能级用户故事 + Given/When/Then 验收标准      | 可执行开发任务（后端/前端/测试拆分） |
| **粒度**         | 一个功能/模块                                  | ≤ 2d 每人                            |
| **用户故事示例** | `As a 用户, I want 用邮箱+密码登录, so that …` | 无用户故事，直接出任务               |
| **验收标准**     | 端到端用户体验验收（Given/When/Then）          | 接口契约 / 组件渲染 / 单测通过       |
| **面向对象**     | 产品经理 / 设计师 / 团队共识                   | 开发工程师 / 测试工程师              |
| **是否重新执行** | 不是重新做，而是**继续做下一步**               | —                                    |

> **关键原则**: 两个 Skill 不重复。`prd-writing` 产出的每个用户故事就是 `story-splitting` 的输入原材料。先有功能级用户故事，再拆开发任务。跳过 prd-writing 直接拆任务，容易遗漏验收标准和边界条件。

### 步骤 5：PRD 评审检查

- [ ] 每个用户故事是否有明确的"谁-要什么-为什么"
- [ ] 验收标准是否可用 Given/When/Then 格式表达
- [ ] 边界条件和异常场景是否覆盖
- [ ] 非功能性需求是否提及
- [ ] Out of Scope 是否明确

## 常用 AI Prompts

### 需求澄清

```
用户需求是：「{原始需求}」
请用 5W1H 列出需要澄清的问题，按重要性排序。
```

### 生成 PRD 初稿

```
请根据以下需求撰写 PRD：{需求描述}
格式要求：
- 包含背景、范围、用户故事、验收标准
- 用户故事用 As a/I want/So that 格式
- 验收标准用 Given/When/Then 格式
```

### 任务拆解

```
请将以下 PRD 拆解为开发任务（粒度：半天到一天）：
{PRD 内容}
按模块分组，标注依赖关系，估算复杂度（S/M/L）。
```

## 补：PRD vs UX 交互规范

PRD 和 UX 交互规范是两个不同层面、服务于不同角色的文档。

| 维度 | PRD（产品需求文档） | UX 交互规范 |
|------|---------------------|-------------|
| 回答的问题 | **做什么**（What） | **怎么做**（How） |
| 典型内容 | 功能清单、用户故事、验收标准、API 端点、数据模型 | 页面交互状态、键盘操作、弹窗行为、菜单切换 |
| 读者 | PM / 干系人 / 测试 | 开发者 / 设计师 |
| 验证方式 | Given/When/Then 验收 | 操作 → 预期响应 |
| 变更频率 | 较低（功能稳定后冻结） | 随实现细节调整 |

### 工作顺序

```
原始需求
    ↓
① PRD（What）→ 明确做什么、不做什么、验收标准
    ↓
② 架构设计 → 技术方案、数据模型、API
    ↓
③ UX 交互规范（How）→ 每个页面的状态流转、键盘操作
    ↓
④ 开发实施
```

### 什么时候需要 UX 交互规范

- 页面有复杂的交互状态（搜索跳转→文档高亮→键盘切换）
- 多角色、多权限下的菜单/页面表现差异
- 需要对弹窗、提示、Loading 等细节做统一约定
- 键盘快捷键和导航路径需要明确文档化

### UX 交互规范生成方式

按页面/交互场景组织，每个页面描述：

 1. 初始状态（初次打开）
 2. 用户操作 → 系统响应（清单式）
 3. 边界情况（空数据、错误、跨页面）
 4. 键盘/快捷键（箭头、ESC）

### 保持同步

- PRD 的功能变更应同步更新 UX 规范
- UX 规范的交互变更不应引入 PRD 未定义的功能
- 两者互为补充，不重叠

## 注意事项 / 陷阱

- ⚠️ **AI 容易编造需求** — AI 可能"脑补"不存在的功能。务必人工审核每一条用户故事是否真实业务需要
- ⚠️ **过度拆解** — 任务粒度过细（< 1 小时）反而增加管理成本。推荐粒度：半天 ~ 2 天
- ⚠️ **忽视非功能性需求** — AI 默认聚焦功能，需要主动提示其补充性能/安全/可访问性
- ⚠️ **需求蔓延** — Out of Scope 要写清楚，AI 容易被"再多加一个小功能"带偏
- ✅ **最佳实践**：PRD 完成后先做一轮 AI 辅助的"找漏洞"（Ask AI to find edge cases）
