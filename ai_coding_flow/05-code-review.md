# 阶段五：Code Review

## 概述

Code Review（代码审查）是保证代码质量、传递团队知识、防止缺陷进入生产环境的最后一道防线。AI 在此阶段的作用是**自动化审查常见问题、安全检查、检查规范一致性**，但**高级语义审查仍依赖人工**。

## 核心交付物

| 交付物 | 说明 | AI 可参与程度 |
|---|---|---|
| Review 评论 | 逐行/逐个函数的审查意见 | 高 — 自动化审查 |
| 安全问题报告 | 安全漏洞标记 | 高 — 自动扫描 |
| 规范检查结果 | 代码风格/命名/模式违规 | 高 — 自动化执行 |
| 变更摘要 | PR 改动的简要说明 | 高 — 自动生成 |
| 最终审批/合并 | Review 完成确认 | 人工 |

## AI 辅助工作流

```
开发者提交 PR / 代码变更
    ↓
AI 自动审查 → 格式/规范/Lint（Pre-commit Hook）
    ↓
AI 自动审查 → 安全扫描（Secret detection / SAST）
    ↓
AI 辅助审查 → 逻辑/设计/可读性（AI Coding Agent）
    ↓
人工 Reviewer 审查 → 重点关注 AI 无法判断的部分
    ↓
开发者根据反馈修改
    ↓
(必要时) AI 二次审查 → 确认问题已修复
    ↓
人工批准 → 合并
```

## AI 审查的覆盖范围

### AI 擅长发现的问题（高置信度）

| 类型 | 示例 | 检测难度 |
|---|---|---|
| **代码规范** | 缩进、命名、import 顺序 | ✅ 易 |
| **常见安全漏洞** | SQL 注入、XSS、硬编码密钥 | ✅ 易 |
| **类型错误** | TypeScript/Python 类型不匹配 | ✅ 易 |
| **未使用变量/import** | Dead code 检测 | ✅ 易 |
| **重复代码** | Copy-paste 检测 | ✅ 中 |
| **函数过长** | 圈复杂度过高 | ✅ 中 |
| **错误处理缺失** | 未处理的 Exception/Rejection | ✅ 中 |

### AI 辅助发现的问题（中置信度 — 需人复核）

| 类型 | 示例 |
|---|---|
| **逻辑缺陷** | 边界条件遗漏、空指针风险 |
| **设计模式偏差** | 该用策略模式的地方用了 if-else 链 |
| **性能问题** | N+1 查询、不必要的重渲染 |
| **可测试性差** | 函数副效应太多、依赖难以 Mock |

### AI 基本无法判断（高风险 — 必须人审）

| 类型 | 原因 |
|---|---|
| **业务逻辑正确性** | AI 不知道真实业务规则 |
| **需求对齐** | AI 不知道 PRD 里的隐性需求 |
| **团队约定** | AI 不知道团队内部约定（除非已记入规范文档） |
| **变更意图合理性** | 这个改动到底"为什么"这么做 |

## 审查分级标准

采用三级制，与常见的 Skill 输出格式一致：

| 级别 | 标记 | 含义 | 处理要求 |
|---|---|---|---|
| 🔴 **Critical** | 必须修复 | 功能性 bug、安全漏洞、数据完整性问题 | 合并前必须修复 |
| 🟡 **Suggestion** | 建议优化 | 可读性、性能、设计模式改进 | 建议修复，非强制 |
| 🟢 **Nice to have** | 可选改进 | 代码风格、注释、命名微调 | 有时间再改 |

## AI Review 提示模版

### 通用 Review
```
请 Review 以下代码变更，按以下格式输出：

## 🔴 Critical
- 描述问题（文件:行号）

## 🟡 Suggestion
- 描述建议

## 🟢 Nice to have
- 描述改进

额外要求：
- 标记每个问题的严重级别
- 说明为什么这是问题
- 给出具体修复建议代码
```

### 安全专项 Review
```
请对以下代码进行安全审计，重点关注：
1. 用户输入是否充分验证和净化
2. SQL/NoSQL 查询是否使用参数化查询
3. 认证/授权逻辑是否正确
4. 是否有敏感信息泄露（密钥、Token、用户数据）
5. 文件操作是否有路径穿越风险
```

### Review 前提交检查清单
```
请帮我检查这个 PR 是否适合提交 Review：
- [ ] 是否有未完成的 TODO/FIXME 注释
- [ ] 是否包含调试用的 print/console.log
- [ ] 是否有注释掉的代码
- [ ] 是否有硬编码的密钥或 Token
- [ ] 是否有未处理的错误路径
- [ ] 测试是否全部通过
```

## PR 描述自动生成

AI 还可以辅助生成 PR 描述，提高 Review 效率：

```
请根据以下 diff 生成 PR 描述：
{diff}

格式：
## 变更内容
- 改了哪些文件、做了什么

## 变更原因
- 为什么做这个改动

## 验证方式
- 本地运行了什么测试

## 风险说明
- 可能影响到哪些模块
```

## 流行工具

| 工具 | 类型 | 说明 |
|---|---|---|
| **Reasonix review** | AI Agent | 内置 review 技能，隔离子 Agent 审查 |
| **CodeRabbit** | AI PR Reviewer | GitHub 集成，自动 Review PR |
| **SonarQube** | 静态分析 | 代码质量 + 安全扫描（本地部署） |
| **Semgrep** | SAST | 自定义规则的静态分析 |
| **Trivy** | 安全扫描 | 依赖漏洞 + IaC 安全 |
| **GitHub CodeQL** | 安全分析 | GitHub 原生，语义级安全分析 |
| **ESLint / Ruff / Clippy** | Linter | 语言级规范检查 |

## 可使用的 Skills

| Skill | 说明 | 什么时候用 |
|---|---|---|
| `code-review` | 代码质量审查 | 提交 PR 前/后 |
| `security-review` | 安全专项审查 | 涉及敏感操作、支付、认证、数据导出等 |

## 注意事项 / 陷阱

- ⚠️ **AI 审查不能替代人审** — AI 漏掉的逻辑错误可能比找出来的还关键。AI Review 是"第一道筛子"，不是最后一道
- ⚠️ **AI 容易产生假阳性** — AI 可能标记"没事找事"的问题（false positive）。Reviewer 需要对 AI 的评论做二次过滤
- ⚠️ **AI 对大型 diff 效果差** — 超过 500 行的 diff，AI 容易遗漏上下文依赖的问题。建议：大 PR 拆小
- ⚠️ **别让 AI 决定是否合并** — 最终合并权限永远在人工手中
- ⚠️ **同源偏差** — 如果代码和 Review 是同一个 AI 生成的，可能发现不了它自己引入的问题。建议：生成用 Reasonix，Review 用其他工具或分步进行
- ✅ **最佳实践**：AI Review 作为 Pre-merge 门禁的一部分（自动+人工），但不过度依赖自动化
