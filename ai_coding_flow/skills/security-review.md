# 安全审查

## Quick Start

1. 列出变更涉及的安全敏感区域
2. 逐项检查：认证/授权/输入验证/数据存储/通信
3. 检查依赖库是否存在已知漏洞
4. 检查配置中是否有硬编码密钥或弱密码
5. 按严重级别输出安全隐患

## 安全审查清单

### 认证与授权
- [ ] 密码是否使用强哈希（bcrypt/argon2）
- [ ] Token 是否安全存储（HTTPOnly Cookie / Secure LocalStorage?）
- [ ] 是否有权限校验（用户不能越权访问他人数据）
- [ ] 会话是否超时失效
- [ ] 是否有暴力破解防护（rate limit / captcha）

### 输入验证
- [ ] 所有用户输入是否经过验证和净化
- [ ] SQL/NoSQL 查询是否使用参数化查询（防注入）
- [ ] 是否对 HTML/JS 输出做转义（防 XSS）
- [ ] 文件上传是否有类型/大小限制
- [ ] API 参数是否有严格的 schema 校验

### 数据安全
- [ ] 敏感数据（密码、Token、PII）是否加密存储
- [ ] 日志中是否可能泄露敏感信息
- [ ] 数据库连接字符串/密钥是否在配置中硬编码

### 依赖安全
- [ ] 是否存在已知 CVE 漏洞的依赖
- [ ] 是否使用了不再维护的库

## 输出格式

| 严重级别 | 问题 | 位置 | 修复建议 |
|---|---|---|---|
| 🔴 Critical | SQL 注入风险 | app/routes.py:42 | 使用参数化查询 |
| 🟡 Suggestion | 无 rate limit | app/routes.py:10 | 添加速率限制中间件 |

## Rules

- 安全问题是 Critical，不降级为 Suggestion
- 所有安全问题必须提供具体的修复建议
- 如果无法确认某项安全风险，标注 ⚠️ 待确认
