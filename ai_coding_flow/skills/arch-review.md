# 技术架构评审

## Quick Start

1. 收集架构设计文档（选型理由、系统图、数据模型、API 设计）
2. 检查技术选型是否匹配需求（功能 + 非功能）
3. 检查架构是否覆盖所有功能模块
4. 识别潜在风险点（单点故障、性能瓶颈、扩展性限制）
5. 列出备选方案对比

## 评审维度

| 维度 | 检查要点 |
|---|---|
| **功能覆盖** | 所有用户故事在架构中是否有对应模块 |
| **非功能需求** | 性能/安全/可用性/扩展性是否已考虑 |
| **技术风险** | 团队是否熟悉所选技术；社区是否活跃 |
| **数据流** | 数据流动方向是否清晰；是否有循环依赖 |
| **耦合度** | 模块间耦合是否合理；是否有过度/不足拆分 |
| **安全** | 认证/授权/加密/防攻击机制是否完整 |

## Checklist

- [ ] 技术选型有明确理由（不只用"流行"）
- [ ] 架构图覆盖所有模块 + 外部依赖
- [ ] 数据模型满足所有用户故事
- [ ] API 设计符合规范（RESTful / GraphQL）
- [ ] 安全机制已提前考虑
- [ ] 扩展路径已预留
- [ ] 已识别至少 3 个潜在风险点

## Output Format

列出架构的优点、风险点（severity: high/med/low）、建议改进项。每个风险点附带缓解策略。
