# 阶段七：运维 / 监控 / 文档

## 概述

运维阶段覆盖系统上线后的持续运营：监控、排障、日志分析、文档维护和知识沉淀。AI 在此阶段的作用是**辅助日志分析、生成排障建议、自动同步知识库**。这个阶段也是下一轮迭代的输入源头。

## 核心交付物

| 交付物 | 说明 | AI 可参与程度 |
|---|---|---|
| 排障记录（Postmortem） | 故障分析文档 | 高 — 从日志/指标生成 |
| 运维手册（Runbook） | 标准操作流程 | 高 — 从排障记录沉淀 |
| 监控 Dashboard | 关键指标可视化 | 中 — 辅助设计 |
| 告警规则 | 异常自动通知 | 中 — 辅助配置 |
| 技术文档 | API 文档、架构说明、README | 高 — 从代码/配置生成 |
| 知识库条目 | 学习笔记沉淀到 wiki | 高 — wiki-ingest Skill |

## AI 辅助工作流

```
系统上线运行
    ↓
监控 + 日志采集
    ↓
(异常发生时) AI 辅助 → 分析日志 + 根因定位
    ↓
(故障修复后) AI 辅助 → 生成 Postmortem 文档
    ↓
AI 辅助 → 更新 Runbook / 运维手册
    ↓
(新知识) AI 辅助 → 整理到知识库（wiki-ingest）
    ↓
(新需求) → 回到阶段 ① PRD —— 循环迭代
```

### 持续迭代闭环

```
运维期发现的问题/需求
    ↓
从运维记录中提取改进项
    ↓
回到 PRD 阶段 → 开始新一轮迭代
    ↓
     ┌─────────────────────────────┐
     │ AI-SDLC 不是线性瀑布流       │
     │ 而是每个阶段都可以回溯的闭环   │
     └─────────────────────────────┘
```

## AI 辅助排障

### 日志分析 prompt

```
以下是生产环境的错误日志：

{粘贴日志}

请分析：
1. 根因是什么（一句话概括）
2. 影响范围（哪些用户/功能受影响）
3. 推荐修复方案（具体步骤）
4. 临时缓解措施（workaround）
```

### 根因定位（多轮对话）

```
Round 1: 用户报告提交订单失败，错误信息是 "500 Internal Server Error"
Round 2: (AI 分析日志) 找到对应的 traceback: KeyError: 'price' 
Round 3: (检查代码) 发现 price 字段在特定商品类型下缺失
Round 4: 根因：商品导入脚本中 B 类型商品缺少 price 字段默认值
```

AI 排障的优势在于能大规模扫描日志，但**根因的"业务含义"需要人工确认**。

### Postmortem 模板

```
# Postmortem: [故障标题]

## 时间线
- 发现时间：
- 恢复时间：
- 持续时间：

## 根因
- 直接原因：
- 深层原因：

## 影响
- 受影响用户数：
- 功能影响范围：
- 经济损失（如有）：

## 修复措施
- 立即修复：
- 长期改进：

## 行动项
- [ ] 行动项 1（Owner, 截止日）
- [ ] 行动项 2（Owner, 截止日）

## 经验教训
- 什么做得好
- 什么可以改进
```

## 文档生成与维护

### README 生成

```
请根据以下项目结构生成 README.md：

{项目目录树}

README 应包含：
- 项目简介
- 快速启动（安装、配置、运行）
- 项目结构说明
- 测试命令
- 部署说明
```

### API 文档更新

```
以下是我的代码变更 diff，请同步更新 OpenAPI 文档：
{diff}
```

### 知识库沉淀（wiki-ingest）

本知识库项目提供了 `wiki-ingest` Skill，可将原始素材整理为 wiki 学习摘要：

```
# 原始素材
{笔记 / 文章 / 对话记录}

# 使用 wiki-ingest Skill 自动处理：
1. 同步到 knowledge_engineering 仓库
2. 搜索相关已有知识
3. 写入结构化的 wiki 摘要
4. 重新索引
5. 检查不存在的链接
```

参见：本项目的 [wiki-ingest Skill](../../README.md) 使用方式。

## 常见排障场景与 AI 提示

### 性能问题
```
当前系统响应时间从 200ms 上升到 2s，
以下是慢查询日志和 CPU/内存监控数据：
{数据}
请分析可能的瓶颈和优化方向。
```

### 依赖/版本问题
```
升级 {包名} 从 v1.0 到 v2.0 后出现以下错误：
{错误信息}
请分析兼容性变更并给出迁移步骤。
```

### 配置问题
```
部署配置如下：
{配置文件}
启动日志：
{日志}
服务启动失败，请分析原因。
```

## 流行工具

| 类别 | 工具 | 说明 |
|---|---|---|
| **日志管理** | ELK / Loki + Grafana | 集中化日志收集和查询 |
| **监控指标** | Prometheus + Grafana | 指标采集 + 可视化 |
| **告警** | Alertmanager / PagerDuty | 告警路由和通知 |
| **APM** | Sentry / Datadog / New Relic | 应用性能监控 |
| **排障** | AI 辅助分析 | Reasonix / Cursor + 日志分析 |
| **文档生成** | Swagger / OpenAPI | API 文档自动化 |
| **知识管理** | Obsidian / Notion / wiki-ingest | 知识沉淀和搜索 |
| **CI 状态** | GitHub / GitLab | 部署状态和 pipeline 可视化 |

## 可使用的 Skills

| Skill | 说明 | 什么时候用 |
|---|---|---|
| `debug-assist` | AI 辅助排障分析 | 收到告警/异常时 |
| `doc-generation` | 自动生成技术文档 | 需要更新 README/API 文档时 |
| `wiki-ingest` | 将素材整理到知识库 | 有新的学习笔记或排障记录需要归档时 |

## 注意事项 / 陷阱

- ⚠️ **AI 分析日志受上下文限制** — 超大日志文件需要先过滤再分析（使用 grep 提取关键行）。AI 无法直接处理数 GB 的原始日志
- ⚠️ **AI 可能误判根因** — 相关性 ≠ 因果性。AI 报告的根因应视为"假设"而非"结论"，需要人工验证
- ⚠️ **不要在 AI 对话中粘贴敏感数据** — 包含用户 PII、密钥、Token 的日志应脱敏后再分析
- ⚠️ **Runbook 需要定期更新** — AI 生成的运维手册需随系统变更同步更新，避免成为"过期文档"
- ⚠️ **知识库是慢价值** — 整理知识库在短期内看不到效果，但长期复用的收益巨大。坚持"写完代码就写文档"的习惯
- ✅ **最佳实践**：每次排障后花 10 分钟让 AI 生成 Postmortem + 更新 Runbook → 用 wiki-ingest 归档到知识库
