# Harness Engineering（驾驭工程）

## 定义

AI 领域的 **Harness Engineering（驾驭工程）**，不是直接训练模型本身，而是设计一套“驾驭、约束、评估、调用、反馈”AI 模型的工程体系，让通用大模型稳定地完成真实任务。

它关注的问题是：**如何把一个通用大模型变成可靠、可控、可持续改进的应用能力。**

`Harness` 有“马具、约束装置、利用”的含义。放到大模型应用里，可以理解为给模型套上一套工程化系统：

- 让模型知道该做什么：`prompt`、system instruction、任务模板
- 让模型能使用工具：函数调用、数据库、搜索、代码执行器、MCP 等
- 让模型行为可控：权限、边界、格式约束、安全策略
- 让结果可评估：测试集、评分器、人工反馈、回归测试
- 让系统可迭代：日志、追踪、失败样本收集、自动优化

## 和相近概念的关系

| 方向 | 关注点 |
|------|--------|
| Prompt Engineering | 怎么问模型 |
| RAG Engineering | 怎么给模型补充知识 |
| Agent Engineering | 怎么让模型规划和调用工具 |
| Eval Engineering | 怎么判断模型好不好 |
| Harness Engineering | 怎么把以上能力组织成稳定系统 |

一句话区分：**Prompt Engineering 更像“写好指令”，Harness Engineering 更像“搭建一整套让 AI 可用的工程系统”。**

## 示例：AI 代码审查助手

只做 Prompt Engineering，可能是写一句：

```text
你是资深代码审查员，请指出潜在 bug。
```

做 Harness Engineering，则会把模型接入完整工作流：

- 自动读取 `git diff`
- 提取相关上下文
- 调用静态分析工具
- 让模型按严重级别输出 findings
- 用历史 bug case 做评测
- 记录误报和漏报
- 对高风险修改触发更严格审查
- 限制模型不能擅自修改无关文件

## 最小化使用

Harness Engineering 不一定一开始就做成复杂平台。最小可用版本可以只包含 5 个部分：

1. **固定任务入口**

   把高频任务写成稳定模板，而不是每次临时聊天。例如：

   ```text
   任务：审查以下 git diff。
   关注：Correctness、Regression、Security、Maintainability。
   输出：只列出确定的问题，按严重程度排序。
   ```

2. **明确输入边界**

   每次只给模型必要上下文，避免把整个项目塞进去。优先输入：

   - 当前任务目标
   - 相关 diff / 文件片段
   - 约束条件
   - 期望输出格式

3. **规定输出结构**

   让输出可读、可检查、可复用。例如代码审查结果固定为：

   ```text
   Severity:
   File:
   Problem:
   Why it matters:
   Suggested fix:
   ```

4. **保留验证动作**

   不把模型回答当最终事实，至少保留一个轻量验证步骤：

   - 运行测试或 lint
   - 对照文档 / 源码确认
   - 对高风险结论人工复核

5. **沉淀失败样本**

   遇到模型误判、漏判、格式错误时，不急着重写整套系统。先把失败样本记下来，用于改模板、改工具输入或补评测用例。

## 最小实践原则

- 先约束输入输出，再增加工具调用。
- 先覆盖高频任务，再追求通用 Agent。
- 先用人工可读日志，再建设复杂观测平台。
- 先做 5 到 10 个真实样本回归，再谈自动优化。
- 对高风险任务保留人工确认，不让模型直接闭环执行。

最小化 Harness 的目标不是“让 AI 自动完成一切”，而是让 AI 在一个清晰、可控、可验证的工作流中稳定发挥作用。
