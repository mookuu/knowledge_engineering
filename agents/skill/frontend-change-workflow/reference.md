# 前端改动参考

## 适用范围

这个 Skill 适用于：
- 页面更新
- 组件修改
- 样式调整
- UI Bug 修复
- 交互行为变更
- 小范围客户端逻辑修复

这个 Skill 不适用于：
- 纯后端改动
- 数据库迁移
- 纯基础设施改动
- 未被明确要求的大型架构重设计

## 项目约定

默认遵循以下约定：
- 优先沿用现有组件模式，不随意设计新的组件 API
- 优先沿用现有样式体系，如 utility classes、CSS modules、design tokens 或主题变量
- 除非确有必要，否则状态应尽量保持局部
- 修改交互式 UI 时要保留可访问性和键盘操作能力
- 最终汇报聚焦用户可见结果、验证方式和剩余风险
- 最终答复默认使用简体中文，除非用户明确要求其他语言

## 推荐验证方式

优先使用与改动最相关的检查：
- Lint：`npm run lint`
- Test：`npm test`
- Build：`npm run build`
- Dev Server：`npm run dev`

如果无法全部运行，优先级建议如下：
1. 先跑与改动区域直接相关的 lint 或 test
2. 如果改动可能影响集成，再补充 build 检查
3. 对视觉或交互型改动，尽量直接做 UI 核对

## 视觉验证规则

在条件允许时，尽量核对：
- 文本内容
- 布局与间距
- hover、active、disabled、error、loading 等状态
- 改动涉及的交互流程
- 受影响区域的基础响应式表现

如果无法直接进行浏览器验证：
- 明确说明未做视觉验证
- 清楚说明剩余风险

## 常见决策规则

### 如果请求主要是视觉改动

- 优先与现有 UI 规范保持一致
- 尽可能直接核对改动后的界面

### 如果请求涉及行为改动

- 重点检查事件处理、状态更新和错误场景
- 如果附近已有测试，优先做有针对性的验证

### 如果请求存在歧义

- 可以根据周边代码和命名做保守推断
- 不要凭空发明新的交互或产品行为

## 常见陷阱

避免：
- 明明只要改一个 prop 或样式，却重做整个组件
- 为一次性场景引入新的模式
- 为了修视觉问题破坏可访问性
- 没有任何验证就直接宣称完成
- 忽略响应式或状态相关回归
