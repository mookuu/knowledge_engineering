# 前端改动流程

## 目的

当任务涉及页面、组件、样式、交互或前端行为改动时，使用这个 Skill，以统一方式完成分析、修改、验证和汇报。

## 何时使用

在以下场景应用这个 Skill：
- 用户要求修改页面、组件、样式、布局或 UI 行为
- 任务涉及组件、页面、样式文件或客户端逻辑
- 用户要求修复前端视觉问题或交互问题
- 改动完成后需要验证用户可见结果

以下场景不要使用：
- 任务只是解释说明，不涉及实际改动
- 工作内容纯属后端、基础设施或数据库
- 已有更具体的项目 Skill 能更准确覆盖该任务

## 快速开始

1. 先明确用户期望的可见结果是什么。
2. 修改前先查看相关组件、页面或样式的现有实现模式。
3. 以满足需求为前提，优先做最小范围的前端改动。
4. 在条件允许时运行有针对性的校验，如 lint、test 或 build。
5. 在可能的情况下直接核对改动后的 UI 效果。
6. 最终用简体中文汇报改动内容、验证方式和剩余风险；如果用户明确要求其他语言，再切换。

## 工作流程

### 第 1 步：收集上下文

- 确认受影响的页面、组件、路由或样式文件。
- 查看附近实现，保持设计与代码风格一致。
- 在新增模式前，优先寻找已有共享组件、设计令牌、工具类或样式约定。

### 第 2 步：规划改动

- 在心里用一句话复述这次 UI 或行为改动的目标。
- 优先扩展现有组件和样式，而不是引入新的抽象。
- 除非需求明确要求，否则保持改动范围收敛。

### 第 3 步：执行改动

- 修改相关前端文件。
- 保持现有结构、样式、命名和状态管理方式一致。
- 非必要时不要顺手做无关清理或重构。

### 第 4 步：执行验证

优先运行与改动最相关的检查，例如：
- `npm run lint`
- `npm test`
- `npm run build`

如果项目提供 dev server 或浏览器验证流程，尽量直接确认改动后的 UI。  
如果无法做完整验证，就执行最有意义的局部检查，并明确说明哪些验证没有运行。

### 第 5 步：核对结果

在条件允许时，确认以下事项：
- UI 是否符合用户要求
- 样式是否与周边页面保持一致
- 是否引入明显回归
- 受影响区域的交互是否仍然正常
- 涉及的响应式表现是否基本合理

### 第 6 步：汇报结果

汇报时应包括：
- 改了什么前端行为或界面
- 跑了哪些检查
- 是否完成了视觉核对
- 是否存在注意事项、后续项或剩余风险

## 检查清单

结束前确认：
- [ ] 改动内容与用户请求一致
- [ ] 遵循了现有前端实现模式
- [ ] 没有引入不必要的抽象或大范围重构
- [ ] 已运行相关验证，或说明了未运行原因
- [ ] 在可能情况下已直接核对受影响 UI
- [ ] 最终答复包含结果、验证情况和剩余风险

## 输出格式

默认使用简体中文输出，代码、命令、路径和标识符保持原样。  
除非用户明确要求英文或其他语言，否则不要切换输出语言。

按以下结构组织最终回复：

1. 简要说明这次 UI 或行为改动
2. 说明做了哪些验证
3. 说明视觉核对是否完成
4. 说明剩余风险、限制或备注
5. 如果自然合适，再补充一个可选下一步

整体语气保持简洁、直接、实用。

## 规则

- 优先做最小范围、可控的改动，不要轻易扩大为大重构。
- 优先复用已有组件、工具方法和样式模式。
- 不要悄悄修改无关 UI。
- 如果需求存在歧义，不要擅自做产品决策。
- 如果无法做视觉验证，要明确说出这一点。
- 当“快速完成”和“与现有前端保持一致”存在冲突时，优先保持一致性。
