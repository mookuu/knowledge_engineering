# 前端改动流程示例

## 示例 1：简单视觉调整

### 用户请求

把登录页主按钮调大一点，但保持现有颜色风格不变。

### 预期行为

- 找到现有登录按钮实现
- 只调整与尺寸相关的样式
- 保留现有颜色令牌和按钮变体
- 如有条件，运行针对性验证
- 核对按钮视觉效果是否正常

### 合格的最终回复

已调整登录页主按钮的尺寸，保留了原有颜色和按钮变体行为。运行了受影响前端代码的 `npm run lint`，并完成了按钮外观的视觉核对。当前剩余风险较低，主要在较小屏幕下周边布局可能还需要进一步确认。

## 示例 2：前端交互 Bug 修复

### 用户请求

修复下拉菜单在选择条目后没有自动关闭的问题。

### 预期行为

- 先检查下拉菜单状态管理逻辑
- 用最小范围修复交互问题
- 检查相关交互是否有回归
- 运行合适的验证
- 在可能情况下直接验证交互流程

### 合格的最终回复

已修复下拉菜单在选择条目后不关闭的问题，通过更新选择处理逻辑来正确重置打开状态。已运行相关验证，并确认改动涉及的交互流程能够正常工作。剩余风险主要在其他未直接验证的菜单入口。

## 示例 3：含糊请求的保守处理

### 用户请求

把这个设置页清理一下。

### 预期行为

- 不要擅自做大范围重设计
- 根据周边代码做小范围整理，必要时再澄清
- 除非用户明确要求，否则不要改变原有行为

### 合格的最终回复

已对设置页做了小范围整理，主要简化了受影响组件的实现，同时保持原有界面行为不变。由于请求没有明确要求产品或布局调整，这次没有扩展成重设计。
