# 瀑布模型 (Waterfall Model)

> **1970 年** Winston Royce 提出。软件工程史上第一个被正式文档化的开发模型。

---

## 核心思想

按固定顺序线性推进，**每个阶段必须 100% 完成并评审通过后，才能进入下一阶段**。如同瀑布流水单向不可逆。

## 阶段划分

```
需求分析 ────→ 系统设计 ────→ 编码实现 ────→ 测试验证 ────→ 部署维护
  │                │               │               │               │
  SRS 文档      架构+详细设计    代码+Code Review  测试报告        上线+运维
```

| 阶段 | 产出 | 评审门控 |
|------|------|---------|
| ① 需求分析 | SRS (软件需求规格说明书) | 需求评审通过 |
| ② 系统设计 | HLD (高层设计) + LLD (详细设计) | 设计评审通过 |
| ③ 编码实现 | 源代码 + 单元测试 | 代码评审通过 |
| ④ 测试验证 | 测试报告 + 缺陷报告 | 测试通过 |
| ⑤ 部署维护 | 部署包 + 运维手册 | 上线审批 |

---

## 特点

| 特点 | 说明 |
|------|------|
| **文档驱动** | 每阶段产出详尽文档，靠文档而非口头沟通传递信息 |
| **阶段门控** | 类似 Stage-Gate——每个阶段结束时设正式评审，通过才放行 |
| **不可逆** | 原则上不回溯；发现需求问题需走变更控制流程（CCB） |
| **一次交付** | 整个项目开发完才交付，用户很晚才看到可运行的软件 |
| **可预测** | 进度和成本在早期即可估算（需求固定的前提下） |

---

## 适用场景

- ✅ 需求极稳定、几乎不会变更（政府/军工/航空）
- ✅ 合同固定价格/范围（Fixed-Price Contract）
- ✅ 强合规审计要求（完整文档链）
- ✅ 团队分散、沟通成本高（文档替代沟通）
- ❌ 互联网产品、SaaS、创业项目
- ❌ 需求不确定或用户说不清想要什么
- ❌ 技术风险未知、需要快速验证

---

## 优缺点

| 优势 | 劣势 |
|------|------|
| 流程清晰、里程碑明确 | 晚期才暴露问题，修复代价极高 |
| 文档完整，便于知识传递和合规审计 | 用户长期看不到可运行软件，反馈极晚 |
| 进度和预算早期可预估 | 对需求变更极度不友好 |
| 适合新人上手 | "大爆炸"式交付风险高 |
| 管理简单（按阶段追踪即可） | 假设需求能在早期被完全理解——这个假设在实践中几乎不成立 |

---

## 经典批判

> "The only thing we know about requirements is that they change."
> — Ivar Jacobson

瀑布的最大问题不是它"错"，而是它的前提（需求不变）在绝大多数项目中不成立。Royce 本人原始的论文也承认这模型有问题，并建议在设计和需求之间加入反馈循环——只是后来的简化版传播成了纯线性。

---

## 与其他模型对比

| vs 模型 | 差异 |
|---------|------|
| vs 敏捷 | 瀑布先设计再编码；敏捷边做边改，短期迭代 |
| vs V模型 | V模型是瀑布的测试增强版，核心逻辑相同 |
| vs 螺旋 | 瀑布不显式管理风险；螺旋每轮以风险为核心 |
| vs 迭代 | 瀑布一次做完；迭代多轮打磨 |
