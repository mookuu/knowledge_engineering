# 精益开发 (Lean Software Development)

> 源自丰田生产系统 (TPS, Taiichi Ohno, 1950s)。Mary & Tom Poppendieck 在 2003 年《Lean Software Development》中将精益原则引入软件领域。

---

## 七大核心原则

| # | 原则 | 含义 | 软件开发中的应用 |
|---|------|------|----------------|
| ① | **消除浪费** | 移除一切不产生客户价值的活动 | 未完成的功能、多余的文档、等待审批 |
| ② | **内建质量** | 不在最后检测质量，在过程中保证 | TDD、CI、Code Review 作为日常而非检查点 |
| ③ | **创造知识** | 开发过程也是学习过程 | Spike、原型、A/B 测试、回顾 |
| ④ | **延迟决策** | 等到最后负责时刻再做决定 | 不提前设计三个月后才需要的架构 |
| ⑤ | **快速交付** | 缩小批次，加速流动 | MVP、小发布、Scrum/Kanban |
| ⑥ | **尊重人** | 一线的人最懂问题在哪 | 自组织团队、信任、心理安全 |
| ⑦ | **整体优化** | 优化全局流动，而非局部效率 | 看价值流而非单个环节的"效率" |

---

## 七种浪费 (Muda) — 软件开发版

| 浪费类型 | 传统制造 | 软件开发对应 |
|---------|---------|-------------|
| **库存** | 堆积的零件 | 未完成的功能/分支（积压的 PR） |
| **等待** | 机器闲置 | 等待 Code Review、等待部署审批 |
| **过度加工** | 不必要的精加工 | 过度设计、黄金镀层（Gold Plating） |
| **缺陷** | 残次品 | Bug、线上事故、返工 |
| **运输** | 材料搬运 | 交接——需求→设计→开发→测试间传递信息 |
| **运动** | 不必要的走动 | 切换上下文、找信息、环境搭建 |
| **过度生产** | 生产超过需求 | 开发没人要的功能 |

## 价值流图 (Value Stream Mapping)

```
需求提出 ──→ 分析 (2d, 等待) ──→ 开发 (3d) ──→ 测试 (3d, 等待)
                                                    │
  实际交付 ←── 部署 (2d, 等待) ←── Code Review (2d) ←┘

  总流量时间: 12 天
  实际增值时间: 3 天 (只有开发)
  效率: 25%

  目标: 压缩等待时间，提高增值占比
```

## 精益 vs 敏捷

| | 精益 | 敏捷 |
|---|------|------|
| **源头** | 制造业 (TPS) | 软件业 (Agile Manifesto) |
| **关注** | 整个价值流的效率 | 团队级的开发过程 |
| **核心问题** | "这个活动产生价值吗？" | "如何快速响应变化？" |
| **范围** | 端到端 (从需求到交付) | 以团队为中心 |
| **可组合** | 精益思想 + Scrum/Kanban 常见 | 敏捷原则 + 精益思想 常见 |

---

## 适用场景

- ✅ 资源有限、需要最大化交付价值的团队
- ✅ 流程审查——发现并消除瓶颈
- ✅ 创业验证阶段（不做不需要的功能）
- ✅ 与 Scrum/Kanban 组合使用效果更佳
- ❌ 精益更适合作为思维框架，而非独立操作方法
