# 迭代与增量模型 (Iterative & Incremental Model)

> 对瀑布的修正：不等到全部完成才给用户看，而是**分次**交付可运行的软件。

---

## 迭代 (Iterative) vs 增量 (Incremental)

两者常被混用，但有本质区别：

| | 迭代式 | 增量式 |
|---|--------|--------|
| **怎么拆** | 按**精化程度**拆 | 按**功能模块**拆 |
| **每轮产出** | 一个更完善的版本（同一功能的质量升级） | 一个新功能块（每次追加功能） |
| **类比** | 画一幅画，先草图 → 线稿 → 上色 → 细化 | 盖房子，先地基 → 框架 → 水电 → 装修 |
| **反馈时机** | 每轮迭代后可调整 | 每块交付后可评估 |
| **风险** | 早期版本不完整但可验证核心逻辑 | 依赖管理——新块可能与旧块冲突 |

在实践中，**绝大多数项目是迭代+增量的混合体**（每个迭代周期既改进已有功能又新增功能）。

---

## 纯迭代模型流程

```
┌────────────┐  ┌────────────┐  ┌────────────┐
│ Iteration 1│→│ Iteration 2│→│ Iteration 3│→ ...
│ 核心功能   │  │ 完善+新功能│  │ 打磨+新功能│
│ 可运行版本 │  │ 可运行版本 │  │ 可运行版本 │
└────────────┘  └────────────┘  └────────────┘
```

每次迭代内部仍可用瀑布流程：
```
需求 → 设计 → 编码 → 测试 → 交付 → 获取反馈 → 下一轮
```

---

## 特点

| 特点 | 说明 |
|------|------|
| **早期反馈** | 每轮迭代后都可向用户演示并获取反馈 |
| **降低风险** | 不等到最后一刻才集成，持续集成降低集成风险 |
| **逐步精化** | 需求可以在后续迭代中调整（比瀑布灵活，比敏捷重一点点） |
| **可运行版本** | 每轮产出的是一个可运行（虽不一定功能完整）的版本 |

---

## 适用场景

- ✅ 系统核心清晰但细节待定 —— 先迭代出核心再扩充
- ✅ 大型系统——通过增量逐步交付降低"大爆炸"风险
- ✅ 新技术需要逐步验证
- ✅ 作为 Scrum/敏捷的底层开发方式
- ❌ 微型项目（拆分的开销不值得）

---

## 优缺点

| 优势 | 劣势 |
|------|------|
| 早期暴露集成问题 | 需求变更管理比纯瀑布复杂 |
| 用户较早看到成果 | 每次迭代仍需要完整的 mini-瀑布，可能感觉"重" |
| 架构可在实践中验证和调整 | 整体架构需要提前设计好（增量可能引入架构矛盾） |

---

## 与敏捷的关系

敏捷的 Scrum / XP 本质上都是迭代增量模型的一种**轻量化实现**——在保留迭代增量收益的同时，用更少的流程、更多的面对面沟通替代重量级文档。
