# V模型 (V-Model)

> 瀑布模型的增强变体，核心改进：**测试不再是最后一步，而是与开发阶段平行对应**。

---

## 核心思想

开发的每个阶段，都同时定义其对应的测试阶段。左侧是"定义/设计"，右侧是"验证/测试"，左右完全对称。

## V 形结构

```
  需求分析 ──────────────────────────────→ 验收测试 (UAT)
     │                                         ↑
     │   对应                                   │
     ↓                                         │
  系统设计 ──────────────────────────────→ 系统测试
     │                                         ↑
     │   对应                                   │
     ↓                                         │
  详细设计 ──────────────────────────────→ 集成测试
     │                                         ↑
     │   对应                                   │
     ↓                                         │
  编码实现 ──────────────────────────────→ 单元测试
     │                                         ↑
     └────────── 时间轴 ──────────────────────→┘

  左侧：验证 (Verification) — "我们在正确地构建产品吗？"
  右侧：确认 (Validation) — "我们在构建正确的产品吗？"
```

| 左侧 (开发) | 右侧 (测试) | 说明 |
|------------|------------|------|
| 需求分析 | 验收测试 (UAT) | 用户需求 → 用户验收 |
| 系统设计 | 系统测试 | 功能/性能/安全 → 全系统验证 |
| 详细设计 | 集成测试 | 模块接口 → 模块间交互验证 |
| 编码实现 | 单元测试 | 代码 → 函数/类级别验证 |

---

## 特点

| 特点 | 说明 |
|------|------|
| **测试先行设计** | 在写需求时就同步规划验收测试；在写设计时就同步规划集成测试 |
| **对称性** | 每个开发活动都有对应的测试活动，不会遗漏验证 |
| **质量内建** | 不把质量寄希望于最后一次性测试 |
| **依然线性** | 与瀑布一样假设阶段顺序执行，对需求变更不友好 |

---

## 适用场景

- ✅ 航空航天 / 汽车电子（ISO 26262 / DO-178C 合规要求）
- ✅ 医疗设备（FDA 要求的完整追责文档链）
- ✅ 安全关键系统（核电、铁路信号）
- ✅ 需求明确且变更成本极高的嵌入式系统
- ❌ 需要快速迭代的 Web/移动应用

---

## 优缺点

| 优势 | 劣势 |
|------|------|
| 测试覆盖系统完整，质量更有保障 | 与瀑布一样的刚性——不适合需求变更 |
| 需求追踪性极强（需求→设计→代码→测试可追溯） | 文档工作量大 |
| 合规审计友好 | 早期投入多，反馈晚 |
| 错误在早期被发现（通过测试设计的并行思考） | 用户很晚才看到可运行软件 |

---

## 与其他模型对比

| vs 模型 | 差异 |
|---------|------|
| vs 瀑布 | V模型多了开发-测试的对称映射；瀑布测试是独立的后置阶段 |
| vs 敏捷 | V模型重文档和追踪；敏捷重快速交付和响应变化 |
| vs TDD | TDD 是编码级的测试先行；V模型是系统级的测试先行设计 |
