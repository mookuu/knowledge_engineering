# 敏捷开发 (Agile Development)

> **2001 年** 17 位软件专家在犹他州雪鸟滑雪场发布了《敏捷宣言》。它不是一种具体方法，而是一套价值观和原则，催生了 Scrum、XP、Kanban 等实践框架。

---

## 敏捷宣言（四项价值观）

| | 左 (更看重) | 右 (但也认可) |
|---|------------|-------------|
| ① | **个体和互动** | 流程和工具 |
| ② | **可工作的软件** | 详尽的文档 |
| ③ | **客户合作** | 合同谈判 |
| ④ | **响应变化** | 遵循计划 |

> 虽然右项也有价值，但我们更重视左项的价值。

---

## 十二项原则 (摘要)

1. 最高优先级：持续交付有价值的软件
2. 欢迎需求变更，即使发生在开发晚期
3. 频繁交付可工作软件（几周到几个月 → 越短越好）
4. 业务人员和开发者每天一起工作
5. 围绕有动力的个体构建项目，给予环境和支持
6. 面对面沟通是最有效的传递信息方式
7. 可工作的软件是衡量进度的**首要**标准
8. 提倡可持续的开发节奏
9. 持续关注技术卓越和良好设计
10. **简洁**—最大化未完成的工作量 — 是根本
11. 最好的架构、需求和设计来自**自组织团队**
12. 团队定期**反思**并调整行为

---

## 敏捷 ≠ 没流程

敏捷不是"不要流程"，而是"流程服务于人，而非人服务于流程"。对比：

| 传统方式 | 敏捷方式 |
|---------|---------|
| 需求 → 需求文档 | 需求 → 用户故事 → 对话 |
| 设计 → 设计文档 | 设计 → 架构刺探 → 演进式设计 |
| 编码 → Code Review (偶尔) | 编码 → TDD + 结对 + 持续集成 |
| 测试 → 独立测试阶段 | 测试 → 每个 Sprint 内完成 |
| 部署 → 手动、低频 | 部署 → 自动化 CI/CD、高频 |

## 敏捷流派/框架

| 框架 | 侧重 | 说明 |
|------|------|------|
| [**Scrum**](scrum.md) | 过程框架 | 角色、事件、工件三件套，最流行的敏捷落地框架 |
| [**Kanban**](kanban.md) | 工作流可视化 | 没有固定迭代，靠 WIP 限制和流管理 |
| [**XP**](xp.md) | 工程实践 | 12 项核心实践，代码质量做到极致 |
| [**Lean / 精益**](lean.md) | 消除浪费 | 源自制造业，快速交付价值 |
| **SAFe** | 规模化敏捷 | 大型企业级框架（有争议：被批"重"到不像敏捷） |
| **LeSS** | 大规模 Scrum | 多个 Scrum 团队协作一个产品的轻量方案 |

---

## 适用场景

- ✅ 需求不确定或可能性变
- ✅ 快速交付 > 完整文档
- ✅ 团队 ≤ 9 人、跨职能、可面对面协作
- ✅ 客户/干系人可频繁参与
- ❌ 需求完全固定且不愿变化的项目
- ❌ 强合规审计需求的系统（可以结合）

## 常见误解

| 误解 | 真相 |
|------|------|
| "敏捷没有文档" | 敏捷有文档，但只写**必要的**——可工作软件优先 |
| "敏捷不计划" | 敏捷持续计划（Sprint Planning），而非一次性大计划 |
| "敏捷就是 Scrum" | Scrum 是最流行的敏捷实现，但不是唯一 |
| "敏捷就是快" | 敏捷追求的是快速**反馈**和快速**适应**，而非单纯快速产出 |
