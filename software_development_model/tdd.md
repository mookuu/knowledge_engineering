# TDD (测试驱动开发 — Test-Driven Development)

> Kent Beck 在 XP 中正式提炼推广。不是"写测试"的方法，而是**用测试驱动设计**的方法。

---

## Red → Green → Refactor 循环

```
 ┌─────────────────────────────────┐
 │                                 │
 │  ❶ RED (红)                     │
 │  写一个失败的测试                 │
 │  确认测试确实失败                 │
 │  (不是语法错，是断言失败)          │
 │          ↓                      │
 │  ❷ GREEN (绿)                   │
 │  写最少的代码让测试通过            │
 │  不求优雅，只求通过                │
 │          ↓                      │
 │  ❸ REFACTOR (重构)              │
 │  消除重复                        │
 │  改进结构                        │
 │  测试必须仍然绿                   │
 │          ↓                      │
 │  回到 ❶ ...                     │
 │          循环 ↓ ──────────────→ │
 └─────────────────────────────────┘
```

**核心纪律**：不在没有失败的测试时写生产代码。

---

## TDD 的三条法则 (Uncle Bob)

1. **除非写一个失败的单元测试，否则不允许写任何生产代码**
2. **只允许写刚好让测试失败所需的测试代码**（编译错误也算失败）
3. **只允许写刚好让失败的测试通过所需的生产代码**

---

## 为什么 TDD 不只是测试

| 不是 TDD | 是 TDD |
|---------|-------|
| 先写代码，后补测试 | 先写测试，后写代码 |
| 测试验证代码 | 测试**驱动设计** |
| 测试是附加品 | 测试是开发流程的一部分 |
| 覆盖率可高可低 | 100% 是新代码的默认要求 |

TDD 的首要收益不是"有测试"，而是：
- **更清晰的接口**（测试即第一个调用者）
- **更少的过度设计**（只写让测试通过的代码）
- **高覆盖率的安全网**（重构底气）

## 三步示例

```python
# ❶ RED — 先写测试
def test_calculate_discount_vip():
    assert calculate_discount(100, is_vip=True) == 80

# (此时 calculate_discount 不存在 → 测试失败 ✅)

# ❷ GREEN — 最少代码让测试通过
def calculate_discount(price, is_vip=False):
    if is_vip:
        return price * 0.8
    return price

# (测试通过 ✅)

# ❸ REFACTOR — 再加普通用户测试
def test_calculate_discount_normal():
    assert calculate_discount(100, is_vip=False) == 100

# (发现逻辑已经对，不用改。重构命名/结构即可)
```

## TDD  vs 传统测试

| 维度 | 传统（Test-Last） | TDD (Test-First) |
|------|:--:|:--:|
| 测试时机 | 代码写完再测 | 代码没写先写测试 |
| 测试目的 | 验证正确性 | 驱动设计 + 验证 |
| 代码可测试性 | 可能很差（事后补救） | 天然可测试（接口即测试） |
| 回归保护 | 取决于事后覆盖率 | 100% 新代码有测试 |
| 设计质量 | 可能耦合过重 | 低耦合（测试要求可隔离） |

---

## 适用场景

- ✅ 逻辑密集的代码（算法、业务规则）
- ✅ 库/框架/SDK 开发
- ✅ 重构前的安全网
- ✅ XP/敏捷团队
- ❌ 一次性脚本/探索性代码
- ❌ UI 布局（更适合快照测试或 E2E）
