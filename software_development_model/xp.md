# 极限编程 (XP — Extreme Programming)

> **1996 年** Kent Beck 在克莱斯勒 C3 项目中实践并总结。XP 把敏捷的工程实践推到极致——**12 项核心实践**，目标是"在最短时间内交付最高质量的软件"。

---

## 价值观与原则

| 价值观 | 含义 |
|--------|------|
| **沟通** | 结对编程、站会、白板——用最直接的方式传递信息 |
| **简单** | 只做今天需要的，不为"可能用到"而设计 |
| **反馈** | TDD → 分钟级反馈；CI → 小时级反馈；用户评审 → 周级反馈 |
| **勇气** | 扔掉没用的代码，对不合理的需求说"不"，承认并修复错误 |
| **尊重** | 团队成员互相关心，信任彼此的能力和承诺 |

## 实践详解

| # | 实践 | 说明 |
|---|------|------|
| ① | **TDD** | Red → Green → Refactor。测试先于代码，见 [tdd.md](tdd.md) |
| ② | **简单设计** | 做最简单的能通过测试的设计。YAGNI (You Ain't Gonna Need It) |
| ③ | **重构** | 持续改进代码结构，保持代码干净 |
| ④ | **隐喻** | 用共享的比喻统一团队对系统的理解（如"购物车"、"流水线"） |
| ⑤ | **结对编程** | 两人一台电脑：Driver 打字，Navigator 思考大局 |
| ⑥ | **代码集体所有** | 任何成员可以改任何代码，无人"拥有"某个模块 |
| ⑦ | **编码标准** | 统一代码风格，代码看起来像一个人写的 |
| ⑧ | **持续集成** | 代码频繁合入主干，每次合入触发自动构建+测试 |
| ⑨ | **小发布** | 每 1-2 周发布一次到生产环境 |
| ⑩ | **可持续节奏** | 不超过 40 小时/周，加班是例外不是常态 |
| ⑪ | **现场客户** | 客户/业务代表在团队现场（或随时可达），即时解答问题 |
| ⑫ | **规划游戏** | 由客户定范围优先级、团队估算成本，共同制定发布计划 |

## 为什么 XP "极端"？

| 好实践 | XP 把它推到极端 |
|--------|----------------|
| 代码评审 | → 变成**结对编程**（实时评审） |
| 测试 | → 变成**TDD**（测试先于代码） |
| 设计 | → 变成**持续重构**（不是一次设计完） |
| 集成 | → 变成**每天多次集成** |
| 简单 | → 变成**只写刚好的代码，绝不多写** |

## 适用场景

- ✅ 需求极其不稳定（创业公司探索期）
- ✅ 对代码质量有极致要求
- ✅ 小团队（≤12人）、集中办公
- ✅ 客户愿意深度参与
- ❌ 团队成员抵触结对编程
- ❌ 分散/远程团队（结对编程难度大）
- ❌ 强合规要求（文档偏少）
