# 螺旋模型 (Spiral Model)

> **1988 年** Barry Boehm 提出。把瀑布的线性流程升级为**风险驱动的迭代循环**，每次循环都明确评估风险。

---

## 核心思想

软件开发不是一条直线，而是一个不断收紧的螺旋。每圈螺旋包含四个象限：

```
            确定目标 ──────→ 风险评估与原型
                ↑                 ↓
                │    (螺旋)        │
                │                 ↓
            规划下一轮 ←──── 开发与测试
```

## 四象限详解

| 象限 | 做什么 | 关键输出 |
|------|--------|---------|
| **① 目标确定** | 明确本轮目标、备选方案、约束条件 | 本轮要实现什么 |
| **② 风险评估** | 识别最大的不确定性，构造原型来降低风险 | 风险清单、原型、可行性分析 |
| **③ 开发与测试** | 实际编码和验证（可以是瀑布也可以是迭代） | 可运行的增量 |
| **④ 规划下一轮** | 评审并规划下一圈螺旋 | Go/No-Go 决策、下轮计划 |

---

## 特点

| 特点 | 说明 |
|------|------|
| **风险驱动** | 每轮螺旋的核心问题是："当前最大的风险是什么？" |
| **渐进式原型** | 高风险区域先做原型确认，低风险区域正常开发 |
| **螺旋收紧** | 随螺旋推进，不确定性逐渐降低，产品逐渐完善 |
| **Go/No-Go 决策点** | 每圈结束都有明确的继续/放弃决策 |

---

## 适用场景

- ✅ 大型、复杂、高风险项目（航天/军事/银行核心系统）
- ✅ 新技术引入（不确定技术方案是否可行）
- ✅ 预算和进度允许有额外的原型探索成本
- ✅ 需要一个灵活框架在"冒险"和"控制"间平衡
- ❌ 中小型项目（螺旋的管理开销超出收益）
- ❌ 快速交付的轻量级项目

---

## 优缺点

| 优势 | 劣势 |
|------|------|
| 风险被显式管理，而非隐式忽略 | 复杂度高，管理成本大 |
| 允许在早期通过原型发现根本性问题 | 需要专业的风险评估能力（中小企业不一定具备） |
| 灵活：每圈可选用不同的开发策略 | 不适合固定价格合同（范围可能随螺旋变化） |
| 客户和开发团队持续对齐 | 螺旋圈数难以提前估算，进度不可预测 |

---

## 与其他模型对比

| vs 模型 | 差异 |
|---------|------|
| vs 瀑布 | 瀑布忽视风险；螺旋以风险为核心 |
| vs 敏捷 | 敏捷靠短迭代+回顾管理风险；螺旋有专门的风险分析象限 |
| vs 迭代 | 迭代只是重复开发；螺旋每次先评估风险再决定迭代内容 |
| vs RUP | RUP 可以看作是螺旋的一种实现（RUP = 迭代 + 用况驱动 + 架构为中心） |
